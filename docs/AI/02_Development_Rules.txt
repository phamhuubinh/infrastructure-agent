# 02 - Development Rules
Các quy tắc dưới đây bắt buộc phải được tuân thủ khi làm việc với repository này.
---
# 1. Mục tiêu
Project hướng đến một AI Agent:
- Đơn giản.
- Stateless.
- Model-Driven.
- Tool-Based.
- Dễ bảo trì.
- Dễ mở rộng.
Ưu tiên:
- MVP.
- Đơn giản.
- Dễ đọc.
- Dễ debug.
- Ít dependency.
Không ưu tiên code "đẹp" nếu làm tăng độ phức tạp.
---
# 2. Kiến trúc là nguồn sự thật
Architecture Decisions là nguồn sự thật của project.
Không tự ý thay đổi:
- Responsibility
- Dependency
- Ownership
- Boundary
Nếu cần thay đổi kiến trúc:
- Giải thích Context.
- Giải thích lý do.
- Phân tích trade-off.
- Phân tích ảnh hưởng.
Chỉ được thay đổi sau khi người dùng chấp thuận.
---
# 3. Không Over-Engineering
Không thêm:
- Interface
- Abstract Class
- Factory
- Strategy
- Repository Pattern
- Event Bus
- Plugin System
- Middleware
- Service Locator
nếu chưa có nhu cầu thực tế.
Ưu tiên giải pháp đơn giản nhất.
---
# 4. Một Component - Một Trách Nhiệm
| Component 	  | Responsibility 	   |
|-----------------|------------------------|
| Reasoning Model | Reasoning 		   |
| Agent 	  | Dispatch 		   |
| Tool 		  | Execute 		   |
| Provider 	  | Access Source of Truth |
| Environment     | Source of Truth 	   |
Không trộn trách nhiệm.
---
# 5. Model là trung tâm
Workflow luôn do Reasoning Model quyết định.
Không được để:
- Agent reasoning.
- Tool reasoning.
- Provider reasoning.
- Collector reasoning.
Reasoning chỉ tồn tại trong Model.
---
# 6. Stateless
Không lưu:
- Tool Result
- Observation
- Runtime Response
- Working Context
Working Context chỉ tồn tại trong một Reasoning Session.
Stable Store chỉ lưu dữ liệu ổn định.
---
# 7. Coding
Ưu tiên:
- Hàm nhỏ.
- Tên rõ ràng.
- Logic đơn giản.
- Dễ đọc.
- Patch nhỏ.
Không tách hàm nếu chưa thực sự cần.
---
# 8. Không Refactor Khi Không Cần
Không tự động:
- đổi tên file
- đổi tên class
- đổi tên hàm
- format toàn bộ project
- di chuyển thư mục
trừ khi được yêu cầu.
---
# 9. Không Thêm Dependency
Ưu tiên Standard Library.
Không thêm:
- Framework
- SDK
- Library
nếu có thể giải quyết bằng thư viện chuẩn.
Nếu bắt buộc:
- Giải thích lý do.
- Chờ chấp thuận.
---
# 10. Giữ Tính Tương Thích
Không:
- phá API
- đổi format dữ liệu
- đổi public interface
trừ khi đã được chấp thuận.
---
# 11. MVP First
Thứ tự ưu tiên:
1. Chạy được.
2. Test được.
3. Review được.
4. Sau đó mới tối ưu.
Không thiết kế cho nhu cầu chưa tồn tại.
---
# 12. Tool
Một Tool chỉ làm đúng một việc.
Không tạo Tool đa năng.
---
# 13. Provider
Provider chỉ truy cập Source of Truth.
Không:
- reasoning
- business validation
- workflow
- state management
Provider chỉ trả dữ liệu cho Tool.
---
# 14. Stable Store
Stable Store không phải runtime path mặc định.
Chỉ dùng cho:
- Snapshot
- Migration
- Cache (khi cần)
Runtime luôn ưu tiên Source of Truth.
---
# 15. Working Mode
Mặc định AI hoạt động ở **Implementation Mode**.
Trong chế độ này:
- Implement đúng design đã được phê duyệt.
- Không redesign kiến trúc.
- Không mở rộng sprint.
- Không tự thêm abstraction.
- Không refactor ngoài phạm vi task.
- Giữ patch nhỏ nhất có thể.
Architecture Discussion chỉ diễn ra khi người dùng yêu cầu rõ ràng.
---
# 16. Implementation
Trong implementation task:
- Chỉ sửa đúng phần được yêu cầu.
- Hiểu repository trước khi sửa.
- Giữ nguyên behavior nếu không được yêu cầu thay đổi.
- Không đoán.
- Không tự tạo API.
- Không tự tạo workflow.
- Không tự thêm feature.
---
# 17. Verification
Sau khi implementation hoàn thành:
1. Review thay đổi.
2. Kiểm tra responsibility.
3. Kiểm tra dependency.
4. Kiểm tra boundary.
5. Chạy affected tests.
6. Nếu cần mới chạy full test suite.
7. Kiểm tra git status.
8. Kiểm tra git diff.
Chỉ commit khi verification hoàn thành.
---
# 18. Definition of Done
Một sprint chỉ hoàn thành khi:
- Code hoàn thành.
- Tests thành công.
- Verification thành công.
- Repository sạch.
- Không phá kiến trúc.
- Một logical change.
- Một commit.
---
# 19. Capability Metadata
Tool capability definitions phải có đúng một nguồn sự thật.
Nguyên tắc:
- Child Tool sở hữu capability của chính nó.
- KnowledgeTool tổng hợp capability metadata.
- Bootstrap lấy capability metadata từ KnowledgeTool.
- PromptBuilder chỉ sử dụng capability metadata từ Bootstrap.
- Không được duplicate capability definitions.
---
# 20. Quy tắc cuối cùng
Luôn ưu tiên:
> Đơn giản > Phức tạp
> MVP > Hoàn hảo
> Rõ ràng > Thông minh
> Chạy được > Thiết kế đẹp
> Patch nhỏ > Refactor lớn
> Kiến trúc ổn định > Tối ưu sớm
Nếu có nghi ngờ:
**Chọn phương án đơn giản hơn.**
---
# 21. Implementation Summary
Implementation Summary
Never report planned work as completed.
Completed work must always be verifiable by:
- git diff
- git status
- test results
