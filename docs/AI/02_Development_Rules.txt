# 02 - Development Rules
Các quy tắc dưới đây bắt buộc phải được tuân thủ khi làm việc với repository này.
---
# 1. Mục tiêu
Project hướng đến một AI Agent:
- Đơn giản.
- Stateless.
- Model-Driven.
- Tool-Based.
- Dễ bảo trì.
- Dễ mở rộng.
Không ưu tiên code "đẹp" nếu làm tăng độ phức tạp.
Luôn ưu tiên:
- MVP.
- Đơn giản.
- Dễ đọc.
- Dễ debug.
- Ít dependency.
---
# 2. Kiến trúc là nguồn sự thật
Không tự ý thay đổi:
- Responsibility.
- Dependency.
- Ownership.
- Boundary.
Nếu cần thay đổi kiến trúc:
Phải giải thích:
- Context.
- Lý do.
- Trade-off.
- Ảnh hưởng.
Sau khi được chấp thuận mới thực hiện.
---
# 3. Không Over-Engineering
Không thêm:
- Interface.
- Abstract Class.
- Factory.
- Strategy.
- Repository Pattern.
- Event Bus.
- Plugin System.
- Middleware.
- Service Locator.
nếu project chưa cần.
Ưu tiên code trực tiếp.
---
# 4. Một Component - Một Trách Nhiệm
## Reasoning Model
Chỉ reasoning.
## Agent
Chỉ dispatch.
## Tool
Chỉ execute.
## Collector
Chỉ collect.
## Stable Store
Chỉ lưu Stable Information.
Không được trộn trách nhiệm.
---
# 5. Model là trung tâm
Workflow luôn do Model quyết định.
Không được để:
- Agent quyết định workflow.
- Tool quyết định workflow.
- Collector quyết định workflow.
Model là nơi duy nhất được reasoning.
---
# 6. Stateless
Không lưu:
- Tool Result
- Observation
- Runtime Response
- Working Context
Working Context chỉ tồn tại trong một phiên reasoning.
Stable Store chỉ lưu dữ liệu ổn định.
---
# 7. Viết Code
Ưu tiên:
- Hàm nhỏ.
- Tên rõ ràng.
- Logic đơn giản.
- Dễ đọc.
Nếu hàm quá dài:
Đề xuất tách.
Không tách nếu chưa cần.
---
# 8. Không Refactor Khi Không Cần
Không:
- đổi tên file
- đổi tên class
- đổi tên hàm
- format toàn bộ project
- di chuyển thư mục
trừ khi được yêu cầu.
---
# 9. Không Thêm Dependency
Ưu tiên thư viện chuẩn của Python.
Không thêm:
- Framework
- SDK
- Library
nếu có thể giải quyết bằng standard library.
Nếu bắt buộc phải thêm dependency:
Phải giải thích lý do.
---
# 10. Giữ Tính Tương Thích
Không:
- phá API
- đổi format dữ liệu
- đổi public interface
trừ khi đã thống nhất.
---
# 11. MVP First
Luôn theo thứ tự:
1. Chạy được.
2. Test được.
3. Review được.
4. Sau đó mới tối ưu.
Không thiết kế cho nhu cầu chưa tồn tại.
---
# 12. Tool
Một Tool chỉ làm một việc.
Ví dụ:
- Inventory Tool
- Command Tool
- Docker Tool
- SSH Tool
Không tạo Tool đa năng.
---
# 13. Collector
Collector chỉ thu thập dữ liệu.
Không:
- reasoning
- validate business
- phân tích dữ liệu
Collector chỉ sinh Stable Information.
---
# 14. Stable Store
Stable Store chỉ lưu dữ liệu thay đổi ít.
Không lưu:
- log
- command output
- observation
- runtime state
---
# 15. Khi Đề Xuất Giải Pháp
Luôn ưu tiên:
Phương án đơn giản nhất.
Nếu có nhiều phương án:
- phân tích ưu điểm
- phân tích nhược điểm
- giải thích trade-off
- đề xuất phương án phù hợp với MVP
---
# 16. Khi Viết Code
Chỉ viết đúng phần được yêu cầu.
Không tự động:
- thêm helper
- thêm TODO
- thêm abstraction
- thêm feature
- refactor phần khác
---
# 17. Không Đoán
Nếu thiếu thông tin:
- Không đoán.
- Không suy diễn.
- Không tự tạo dữ liệu.
- Không tự tạo API.
- Không tự tạo workflow.
Đọc repository trước.
Nếu vẫn thiếu thì hỏi.
---
# 18. Verification
Sau mỗi thay đổi cần review:
- Responsibility.
- Dependency.
- Boundary.
Sau đó chạy:
```bash
git status
git diff
ruff check src tests
python -m pytest
```
Nếu chỉ sửa nhỏ:
Có thể chạy test liên quan trước.
Sau đó mới chạy toàn bộ test.
---
# 19. Definition of Done
Một sprint chỉ hoàn thành khi:
- Code hoàn thành.
- Test thành công.
- Verification thành công.
- Repository sạch.
- Không phá kiến trúc.
---
# 20. Quy tắc cuối cùng
Mọi quyết định đều phải ưu tiên:
> Đơn giản > Phức tạp
> MVP > Hoàn hảo
> Rõ ràng > Thông minh
> Có thể chạy > Thiết kế đẹp
> Kiến trúc ổn định > Tối ưu sớm
Nếu có nghi ngờ:
Chọn phương án đơn giản hơn.
