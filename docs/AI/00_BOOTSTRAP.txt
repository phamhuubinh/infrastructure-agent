# Mục tiêu
Project xây dựng một AI Agent theo kiến trúc **Model-Driven**.
Mục tiêu:
- Đơn giản.
- Stateless.
- Ít token.
- Dễ bảo trì.
- Dễ mở rộng.
- Không over-engineering.
The MVP is complete.
The project is now in the Platform Evolution phase.
Development is benchmark-driven.
Every significant change must be validated through automated benchmarks before being considered complete.

# Triết lý
Reasoning và Execution là hai trách nhiệm hoàn toàn khác nhau.
## Reasoning Model
Chịu trách nhiệm:
- Hiểu yêu cầu người dùng.
- Lập kế hoạch.
- Quyết định cần thông tin gì.
- Quyết định tool nào cần gọi.
- Phân tích kết quả.
- Sinh Final Response.
Không:
- Thu thập dữ liệu.
- Thao tác môi trường.
- Truy cập trực tiếp hệ điều hành.
- Truy cập Stable Store.
---
## Agent
Agent chỉ là Dispatcher.
Chịu trách nhiệm:
- Nhận yêu cầu từ Model.
- Gọi Tool.
- Trả kết quả về Model.
Không:
- Reasoning.
- Business Logic.
- Quyết định workflow.
- Phân tích dữ liệu.
---
## Tool
Tool chỉ làm đúng một nhiệm vụ.
Ví dụ:
- Đọc inventory.
- Chạy command.
- Lấy log.
- Truy cập API.
Tool:
- Không reasoning.
- Không biết workflow.
- Không quyết định bước tiếp theo.
---
## Provider
Provider chỉ truy cập Source of Truth.
Provider:
- Không reasoning.
- Không biết Model.
- Không biết workflow.
- Chỉ trả dữ liệu cho Tool.
---
# Stateless
Reasoning remains stateless.
A lightweight session memory may exist outside the reasoning loop.
Session memory stores summarized facts only.
It never stores:
- raw observations
- tool results
- runtime responses
- conversation history
The model decides whether summarized knowledge is sufficient or fresh evidence should be collected.
Working Context chỉ tồn tại trong phiên hiện tại.
Không lưu:
- Observation
- Tool Result
- Runtime Response
- Working Context
Stable Store chỉ lưu dữ liệu ổn định.
---
# Stable Information
Stable Information là dữ liệu thay đổi ít.
Ví dụ:
- Hardware
- Operating System
- SSH
- Docker
- VMware
- Installed Software
- Repository
Stable Information có thể được lưu dưới dạng snapshot khi cần.
Model chỉ truy cập dữ liệu thông qua Tool.
---
# Working Information
Working Information gồm:
- Tool Result
- API Response
- Command Output
- Observation
Working Information exists only during a reasoning session.
After Final Response,
raw observations,
tool outputs,
runtime state
must be discarded.
Only summarized session facts may remain outside the reasoning loop.

# Vai trò của AI
Người dùng sở hữu kiến trúc của dự án.
Mặc định, AI hoạt động ở **Implementation Mode**.
AI có trách nhiệm:
- Hiểu kiến trúc hiện tại.
- Tuân thủ Architecture Decisions.
- Hiểu repository trước khi thay đổi.
- Thực hiện đúng design đã được phê duyệt.
- Giữ patch nhỏ nhất có thể.
- Giữ nguyên hành vi hiện có nếu không được yêu cầu thay đổi.
- Chạy các affected tests.
- Sửa lỗi implementation nếu phát sinh.
AI không được:
- Tự thay đổi kiến trúc.
- Tự mở rộng phạm vi công việc.
- Tự thêm abstraction.
- Tự thêm dependency.
- Refactor ngoài phạm vi task.
- Đề xuất kiến trúc mới khi chưa được yêu cầu.
Architecture Discussion Mode chỉ được sử dụng khi người dùng yêu cầu rõ ràng.
Benchmark-Driven Development
When implementing new functionality,
prefer writing or updating benchmark scenarios before expanding the implementation.
New capabilities should exist because benchmark scenarios require additional evidence,
not because they appear generally useful.

# Quy tắc làm việc
Mỗi task thực hiện theo quy trình sau:
1. Đọc repository.
2. Hiểu design đã được phê duyệt.
3. Xác định phạm vi thay đổi.
4. Lập implementation plan.
5. Hiển thị diff.
6. Chờ approval.
7. Thực hiện thay đổi.
8. Run affected tests.
9. Run affected benchmarks when reasoning behaviour may change.
10. Fix regressions.
11. Commit.
12. Stop.
Mỗi sprint:
- Một logical change.
- Một review.
- Một test cycle.
- Một commit.
Không được:
- Đoán.
- Suy diễn.
- Thay đổi kiến trúc.
- Mở rộng sprint.
- Thực hiện nhiều feature trong một sprint.
---
# Khi trả lời
Ưu tiên:
1. Hiểu vấn đề.
2. Xác nhận phạm vi.
3. Thực hiện đúng design đã được phê duyệt.
4. Giữ patch nhỏ nhất.
5. Chạy affected tests.
6. Dừng sau khi hoàn thành sprint.
Nếu người dùng yêu cầu kiến trúc:
1. Phân tích.
2. Trade-off.
3. Đề xuất phương án.
4. Chờ quyết định.
Không tự động:
- Refactor.
- Đổi tên.
- Thêm helper.
- Thêm abstraction.
- Thêm tính năng.
- Thực hiện công việc ngoài phạm vi được giao.
---
Development Priorities

4. Benchmark Expansion
