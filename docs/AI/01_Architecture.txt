# 01 - Architecture
## Tổng quan
Hệ thống được thiết kế theo kiến trúc **Model-Driven**.
Reasoning chỉ tồn tại trong Model.
Mọi thành phần còn lại chỉ thực hiện đúng trách nhiệm của mình.
```text
                User
                  │
                  	▼
          Reasoning Model
                  │
                  	▼
                Agent
                  │
                  	▼
          Tool Dispatcher
                  │
     ┌────────────┴────────────┐
     ▼                         		▼
Knowledge Tool           Runtime Tool
     │                         │
     ▼                         		▼
Stable Store             Environment
                          (Linux/API/...)
```
---
# Kiến trúc tổng thể
Hệ thống gồm các thành phần sau:
- Reasoning Model
- Agent
- Tool
- Stable Store
- Collector
- Environment
Mỗi thành phần chỉ có một trách nhiệm.
---
# 1. Reasoning Model
Reasoning Model là trung tâm của toàn bộ hệ thống.
Model chịu trách nhiệm:
- hiểu yêu cầu người dùng
- lập kế hoạch
- quyết định cần dữ liệu gì
- quyết định tool nào cần gọi
- phân tích kết quả
- quyết định khi nào kết thúc
- sinh Final Response
Model không:
- truy cập OS
- đọc file
- gọi API
- biết Tool hoạt động như thế nào
- biết dữ liệu được lưu ở đâu
Model chỉ giao tiếp với Agent.
---
# 2. Agent
Agent là Dispatcher.
Agent chịu trách nhiệm:
- nhận request từ Model
- validate request
- gọi Tool tương ứng
- trả Tool Result về Model
- lặp lại cho tới khi Model trả Final Response
Agent không:
- reasoning
- phân tích dữ liệu
- quyết định workflow
- lưu trạng thái
Agent càng nhỏ càng tốt.
---
# 3. Tool
Tool là lớp thực thi.
Một Tool chỉ làm một việc.
Ví dụ:
- Inventory Tool
- Command Tool
- SSH Tool
- Docker Tool
- VMware Tool
Tool:
- đọc dữ liệu
- chạy command
- gọi API
- trả dữ liệu
Tool không:
- reasoning
- quyết định bước tiếp theo
- biết mục tiêu cuối cùng
---
# 4. Stable Store
Stable Store lưu dữ liệu thay đổi ít.
Ví dụ:
- Hardware
- Operating System
- Docker Inventory
- Installed Software
- SSH
- Repository Information
Stable Store tồn tại qua nhiều session.
Stable Store chỉ được truy cập thông qua Tool.
Model không truy cập trực tiếp Stable Store.
---
# 5. Collector
Collector chịu trách nhiệm tạo Stable Store.
Collector:
- thu thập dữ liệu
- chuẩn hóa dữ liệu
- cập nhật Stable Store
Collector không:
- reasoning
- biết Model
- biết Agent
Collector có thể chạy:
- theo lịch
- theo yêu cầu
- khi inventory thay đổi
---
# 6. Runtime Tool
Một số dữ liệu không nên lưu.
Ví dụ:
- ps
- top
- ping
- curl
- docker logs
- journalctl
Những dữ liệu này được Runtime Tool lấy trực tiếp từ Environment.
Không lưu vào Stable Store.
---
# 7. Working Context
Working Context chỉ tồn tại trong một phiên reasoning.
Bao gồm:
- Tool Result
- Observation
- API Response
- Command Output
Working Context bị hủy sau khi sinh Final Response.
Không lưu vào Stable Store.
---
# 8. Luồng hoạt động
```text
User
    │
    ▼
Reasoning Model
    │
    ▼
Agent
    │
    ▼
Tool
    │
    ▼
Environment / Stable Store
    │
    ▼
Tool Result
    │
    ▼
Agent
    │
    ▼
Reasoning Model
    │
    ▼
Final Response
```
Nếu cần nhiều Tool:
```text
Model
 ↓
Tool A
 ↓
Model
 ↓
Tool B
 ↓
Model
 ↓
Tool C
 ↓
Model
 ↓
Final Response
```
Model luôn là thành phần quyết định bước tiếp theo.
---
# 9. Dependency
Dependency chỉ đi theo một chiều.
```text
Reasoning Model
        │
        ▼
Agent
        │
        ▼
Tool
        │
        ├────────► Stable Store
        │
        └────────► Environment
```
Không được phụ thuộc ngược.
---
# 10. Responsibility
| Component    | Responsibility 	  |
|--------------|--------------------------|
| Model        | Reasoning 		  |
| Agent        | Dispatch                 |
| Tool         | Execute                  |
| Collector    | Collect                  |
| Stable Store | Store Stable Information |
| Environment  | Thực thi thực tế        |
---
# 11. Nguyên tắc mở rộng
Khi cần thêm nguồn dữ liệu mới:
Ví dụ:
- Kubernetes
- GitHub
- Jira
- AWS
- Azure
Chỉ cần thêm Tool hoặc Collector mới.
Không thay đổi:
- Model
- Agent
- Kiến trúc tổng thể
---
# 12. Mục tiêu cuối cùng
Kiến trúc phải luôn giữ được:
- Stateless
- Model-Driven
- Tool-Based
- Low Coupling
- High Cohesion
- MVP First
- Simple over Complex
Mọi quyết định thiết kế đều phải phục vụ các nguyên tắc trên.
