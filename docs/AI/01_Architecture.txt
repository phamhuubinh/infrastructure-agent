# 01 - Architecture
## Tổng quan
Hệ thống được thiết kế theo kiến trúc **Model-Driven**.
Reasoning chỉ tồn tại trong Reasoning Model.
Mọi thành phần còn lại chỉ thực hiện đúng trách nhiệm của mình.
Reasoning Model
        │
            ▼
Agent
        │
            ▼
KnowledgeTool (Tool tổng)
        │
        ├───────────────┐
        │               │
            ▼                      ▼
LinuxTool      ZabbixTool      VMwareTool      DockerTool
        │               │                │               │
            ▼                      ▼                       ▼                      ▼
     Linux         Zabbix API        vCenter API     Docker Engine
KnowledgeTool là Tool duy nhất được Agent gọi.
KnowledgeTool là Tool tổng duy nhất trong runtime.
Model không biết sự tồn tại của LinuxTool, DockerTool, VMwareTool...
KnowledgeTool chịu trách nhiệm định tuyến request tới Tool con phù hợp.
Tool con không biết Model.
Model chỉ biết KnowledgeTool.
KnowledgeTool chịu trách nhiệm:
- chọn Tool con
- gọi Tool con
- trả ToolResult
KnowledgeTool không truy cập trực tiếp Environment.
## Capability Discovery
KnowledgeTool owns the capability catalog exposed to the Reasoning Model.
KnowledgeTool discovers capabilities from child Tools.
Child Tools are the single source of truth for their own capabilities.
The Reasoning Model never discovers capabilities directly from child Tools.
PromptBuilder never imports or depends on child Tools.
Tool con chịu trách nhiệm duy nhất cho domain của mình.
*Provider là tùy chọn.
Tool có thể truy cập trực tiếp Environment nếu không cần Provider.
---
# Kiến trúc tổng thể
Hệ thống gồm các thành phần:
- Reasoning Model
- Agent
- Tool
- Provider (tùy chọn)
- Environment
Mỗi thành phần chỉ có một trách nhiệm.
---
# 1. Reasoning Model
Reasoning Model là trung tâm của toàn bộ hệ thống.
Model chịu trách nhiệm:
- hiểu yêu cầu người dùng
- lập kế hoạch
- quyết định cần dữ liệu gì
- quyết định Tool nào cần gọi
- phân tích Tool Result
- quyết định bước tiếp theo
- sinh Final Response
Model không:
- đọc file
- chạy command
- gọi API
- truy cập hệ điều hành
- biết dữ liệu được lấy như thế nào
Model chỉ giao tiếp với Agent.
---
# 2. Agent
Agent là Dispatcher.
Agent chịu trách nhiệm:
- nhận request từ Model
- validate request
- gọi Tool tương ứng
- trả Tool Result về Model
- lặp lại cho tới khi Model sinh Final Response
Agent không:
- reasoning
- phân tích dữ liệu
- quyết định workflow
- lưu trạng thái
Agent càng nhỏ càng tốt.
---
# 3. Tool
Tool là lớp thực thi.
Một Tool chỉ thực hiện một nhiệm vụ.
Ví dụ:
- Knowledge Tool
- Command Tool
- SSH Tool
- Docker Tool
- VMware Tool
Tool chịu trách nhiệm:
- nhận request
- gọi Provider hoặc Environment
- trả Tool Result
Tool không:
- reasoning
- quyết định workflow
- lưu trạng thái
- biết mục tiêu cuối cùng
Một số Tool có thể truy cập Provider.
Một số Tool có thể truy cập trực tiếp Environment nếu không cần Provider.
Ví dụ:
KnowledgeTool
        │
            ▼
LinuxTool
        │
        ├──────────────► Linux
        │
        └──────────────► LinuxProvider (optional)
                               	│
                               		▼
                              Linux
---
# 4. Tool Implementation
Tool được chia theo từng domain.
Ví dụ:
- Linux Tool
- Zabbix Tool
- VMware Tool
- Docker Tool
- SSH Tool
Mỗi Tool chỉ chịu trách nhiệm cho domain của mình.
Tool có thể:
- truy cập trực tiếp Environment;
- hoặc thông qua Provider nếu việc truy cập đủ phức tạp.
Tool không:
- reasoning;
- quyết định workflow;
- lưu trạng thái.
---
# 5. Provider
Provider là lớp tùy chọn.
Provider được tạo khi:
- logic truy cập Source of Truth đủ phức tạp;
- cần tái sử dụng;
- cần chuẩn hóa nhiều API.
Nếu Tool chỉ cần thực hiện một vài command hoặc một vài API đơn giản thì Tool có thể truy cập trực tiếp Environment.
Mỗi Provider chỉ biết cách truy cập một Source of Truth cụ thể.
Ví dụ:
- Linux Provider
- Docker Provider
- SSH Provider
- VMware Provider
- Zabbix Provider
- Grafana Provider
- Graylog Provider
Provider chịu trách nhiệm:
- truy cập Source of Truth;
- chuẩn hóa dữ liệu;
- trả dữ liệu cho Tool.
Provider không:
- reasoning;
- quyết định workflow;
- biết Model;
- biết Agent;
- lưu trạng thái.
---
# 6. Environment
Environment là Source of Truth của hệ thống.
Ví dụ:
- Linux
- Docker Engine
- Kubernetes
- VMware API
- Zabbix API
- Grafana API
- Graylog API
Tool luôn ưu tiên đọc trực tiếp từ Source of Truth.
Environment không biết:
- Model
- Agent
- Tool
---
# 7. Working Context
Working Context chỉ tồn tại trong một Reasoning Session.
Bao gồm:
- Tool Result
- Observation
- API Response
- Command Output
Working Context bị hủy sau khi sinh Final Response.
Không lưu qua các phiên.
---

---
# 9. Dependency
Dependency chỉ đi theo một chiều.
Reasoning Model
        │
            ▼
Agent
        │
            ▼
Tool
        │
        ├────────► Provider
        │              │
        │              ▼
        └────────► Environment
Không được phụ thuộc ngược.
---
# 10. Responsibility
| Component   | Responsibility  |
|-------------|-----------------|
| Model       | Reasoning       |
| Agent       | Dispatch        |
| Tool        | Execute 	|
| Provider    | Access Source   |
| Environment | Source of Truth |
Mỗi component chỉ có một responsibility.
---
# 11. Nguyên tắc mở rộng
Khi cần hỗ trợ hệ thống mới:
Ví dụ:
- Kubernetes
- GitHub
- Jira
- AWS
- Azure
Chỉ cần:
- thêm Tool mới;
- hoặc thêm Provider nếu Tool cần tách logic truy cập dữ liệu.
Không cần thay đổi:
- Model
- Agent
- Kiến trúc tổng thể
---
# 12. Mục tiêu cuối cùng
Kiến trúc phải luôn giữ được:
- Model-Driven
- Stateless
- Tool-Based
- Source of Truth
- Low Coupling
- High Cohesion
- MVP First
- Simple over Complex
Mọi quyết định thiết kế đều phải phục vụ các nguyên tắc trên.
