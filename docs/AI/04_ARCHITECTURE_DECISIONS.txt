# 04 - Architecture Decisions
Tài liệu này lưu các quyết định kiến trúc của project.
Mỗi quyết định phải gồm:
- Decision
- Context
- Reason
- Consequence
Không ghi:
- TODO
- Roadmap
- Implementation Detail
---
# AD-001
## Decision
Reasoning chỉ thuộc về Reasoning Model.
## Context
Project được thiết kế theo kiến trúc Model-Driven.
## Reason
Tách biệt hoàn toàn Reasoning và Execution.
## Consequence
- Agent không reasoning.
- Tool không reasoning.
- Provider không reasoning.
---
# AD-002
## Decision
Agent chỉ là Dispatcher.
## Context
Agent nằm giữa Model và Tool.
## Reason
Giảm Business Logic.
Giữ Agent đơn giản.
## Consequence
Agent chỉ:
- nhận request
- gọi Tool
- trả Tool Result
Không:
- phân tích dữ liệu
- quyết định workflow
- lưu trạng thái
---
# AD-003
## Decision
Một Tool chỉ thực hiện một nhiệm vụ.
## Context
Project sử dụng Tool-Based Architecture.
## Reason
Giảm coupling.
Dễ mở rộng.
## Consequence
Một Tool chỉ:
- đọc dữ liệu
- chạy command
- gọi API
Không tạo Tool đa năng.
---
# AD-004
## Decision
Provider chỉ truy cập Source of Truth.
## Context
Provider là lớp kết nối giữa Tool và hệ thống thực.
## Reason
Tách biệt việc truy cập dữ liệu khỏi Reasoning và Tool Dispatch.
## Consequence
Provider:
- không reasoning
- không biết Model
- không biết Agent
- không biết workflow
Provider chỉ truy cập Source of Truth và trả dữ liệu.
---
# AD-005
## Decision
Runtime luôn ưu tiên Source of Truth.
## Context
Nguồn dữ liệu đáng tin cậy nhất là chính Environment.
## Reason
Đảm bảo dữ liệu luôn mới.
Giảm phụ thuộc vào dữ liệu snapshot.
## Consequence
Tool luôn ưu tiên:
Environment
↓
Provider
↓
Tool
Stable Store (nếu tồn tại) chỉ là snapshot hoặc cơ chế migration.
Không còn là runtime path mặc định.
---
# AD-006
## Decision
Working Context không được lưu.
## Context
Reasoning Session phải Stateless.
## Reason
Giảm coupling.
Tránh dữ liệu cũ ảnh hưởng phiên mới.
## Consequence
Không lưu:
- Tool Result
- Observation
- API Response
- Command Output
Working Context bị hủy sau Final Response.
---
# AD-007
## Decision
Workflow luôn do Model quyết định.
## Context
Model là thành phần duy nhất reasoning.
## Reason
Giữ Agent và Tool không chứa Business Logic.
## Consequence
Agent không quyết định:
- gọi Tool nào
- gọi khi nào
- dừng khi nào
Toàn bộ workflow thuộc về Model.
---
# AD-008
## Decision
Model không truy cập trực tiếp Environment.
## Context
Environment có thể thay đổi.
## Reason
Giảm coupling.
Ẩn implementation.
## Consequence
Model chỉ giao tiếp với Agent.
Agent chỉ giao tiếp với Tool.
Tool mới được thao tác Environment.
---
# AD-009
## Decision
Model không truy cập trực tiếp Source of Truth.
## Context
Model không được phụ thuộc implementation.
## Reason
Cho phép thay đổi Provider hoặc Environment mà không ảnh hưởng Model.
## Consequence
Mọi truy cập dữ liệu đều phải thông qua Tool.
Model không biết dữ liệu đến từ:
- Linux
- Docker
- VMware
- Zabbix
- API
---
# AD-010
## Decision
Dependency chỉ đi theo một chiều.
## Context
Project áp dụng kiến trúc phân tầng.
## Reason
Giảm phụ thuộc ngược.
Dễ thay thế từng thành phần.
## Consequence
```text
Reasoning Model
        │
            ▼
Agent
        │
            ▼
Tool
        │
            ▼
Provider
        │
            ▼
Environment
```
Không được phụ thuộc ngược.

# AD-012
## Decision
Một Component chỉ có một Responsibility.
## Context
Toàn bộ project áp dụng nguyên tắc Single Responsibility.
## Reason
Dễ đọc.
Dễ debug.
Dễ thay thế.
## Consequence
| Component   | Responsibility  |
|-------------|-----------------|
| Model       | Reasoning       |
| Agent       | Dispatch        |
| Tool 	      | Execute 	|
| Provider    | Access Source   |
| Environment | Source of Truth |
Không chuyển Responsibility giữa các Component.

# AD-013
## Decision
Không lưu Business Workflow trong hệ thống.
## Context
Workflow được sinh động bởi Model.
## Reason
Giữ hệ thống Stateless.
## Consequence
Agent và Tool không biết:
- bước trước
- bước sau
- mục tiêu cuối cùng
Chúng chỉ xử lý request hiện tại.
---
# AD-014
## Decision
Mọi thay đổi kiến trúc phải ưu tiên sự đơn giản.
## Context
Project hướng tới khả năng bảo trì lâu dài.
## Reason
Độ phức tạp tăng nhanh hơn số lượng tính năng.
## Consequence
Nếu có nhiều phương án:
1. Chọn phương án đơn giản hơn.
2. Chọn ít abstraction hơn.
3. Chọn ít dependency hơn.
4. Chỉ mở rộng khi có nhu cầu thực tế.
---
# AD-015
## Decision
Repository là nguồn sự thật duy nhất.
## Context
AI có thể có trí nhớ hoặc suy luận sai.
## Reason
Đảm bảo mọi thay đổi đều dựa trên trạng thái thực tế của project.
## Consequence
Trước khi:
- thiết kế
- sửa code
- review
- refactor
AI phải đọc repository liên quan.
Nếu repository khác với trí nhớ hoặc suy luận, luôn ưu tiên repository.
# AD-016
## Decision
KnowledgeTool là Tool tổng.
## Context
Project hỗ trợ nhiều domain.
## Reason
Model không nên biết Linux, Docker, VMware...
## Consequence
Agent chỉ gọi KnowledgeTool.
KnowledgeTool route tới Tool con.
Tool con không biết Agent.
KnowledgeTool không truy cập trực tiếp Environment.
KnowledgeTool chỉ chịu trách nhiệm định tuyến request tới Tool con.
Tool con chịu trách nhiệm truy cập Environment hoặc Provider.
## AD-022 Tool Capability Ownership
Child Tools own their capability definitions.
KnowledgeTool aggregates capabilities from child Tools.
Bootstrap is generated from KnowledgeTool.
PromptBuilder consumes Bootstrap only.
PromptBuilder must never hardcode Tool capabilities.
Reason:
Capability definitions must exist in exactly one place.
This avoids duplicated capability lists and keeps the protocol synchronized automatically.
KnowledgeTool is the only component allowed to aggregate capability metadata from child Tools.
