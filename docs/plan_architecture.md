# Plan Architecture
---
# Purpose
Plan Architecture defines how execution plans are represented, validated, and executed.
A Plan describes how an agent achieves a Goal through an ordered sequence of executable Steps while satisfying defined Constraints.
Implementation details belong to component specifications.
---
# Responsibilities
Plan Architecture is responsible for:
* representing execution plans;
* organizing executable steps;
* defining execution constraints;
* supporting plan validation;
* supporting execution monitoring;
* supporting replanning.
Plan Architecture must not:
* perform planning;
* execute actions;
* perform Runtime operations.
---
# Architectural Principles
The following principles shall always hold:
* A Plan is generated by the Planner.
* A Plan is executed by the Executor.
* A Plan never executes itself.
* Every Step belongs to exactly one Plan.
* Plans evolve through replanning rather than direct mutation.
---
# Core Concepts
## Plan
A Plan represents an ordered strategy for achieving a Goal.
A Plan consists of:
* Goal;
* Steps;
* Constraints;
* Priority;
* Status;
* Metadata.
---
## Goal
A Goal defines the desired outcome of the Plan.
Goals originate from the Knowledge Model.
---
## Step
A Step is the smallest executable unit within a Plan.
Each Step represents one Action together with its execution conditions.
---
## Action
An Action represents an abstract executable operation.
Concrete execution is performed by the Executor.
---
## Constraint
Constraints define the conditions under which a Plan may execute.
Typical constraints include:
* resource limits;
* safety requirements;
* environmental conditions.
---
## Preconditions
Preconditions define the conditions that must be satisfied before a Step may execute.
---
## Postconditions
Postconditions describe the expected state after successful execution.
---
## Expected Result
Expected Results describe the intended outcome of a completed Step.
---
## Retry Policy
Retry Policy defines how execution failures may be retried.
---
## Timeout
Timeout defines the maximum execution duration for a Step.
---
## Execution Mode
Execution Mode determines whether Steps execute:
* Sequentially
* In Parallel
---
## Plan Identity
Each Plan owns:
* Plan ID
* Version
Each Step owns:
* Step ID
These identifiers support tracking and lifecycle management.
---
# Plan Lifecycle
A Plan progresses through the following lifecycle:
* Draft
* Validated
* Executing
* Completed
* Failed
---
# Step Lifecycle
Each Step progresses through:
* Pending
* In Progress
* Completed
* Failed
---
# Execution Flow
```text
Knowledge Model
        ↓
Planner
        ↓
Plan
        ↓
Validation
        ↓
Executor
        ↓
Step Execution
        ↓
Execution Feedback
        ↓
Knowledge Model
```
Execution feedback may trigger replanning.
---
# Replanning
Replanning occurs whenever execution feedback invalidates the current Plan.
The Planner generates a revised Plan using updated knowledge.
---
# Relationships
## Planner
Planner creates and validates Plans.
Planner never executes Plans.
---
## Executor
Executor executes Steps defined by the Plan.
Executor never modifies planning decisions.
---
## Knowledge Model
Knowledge Model provides context for:
* Goals;
* Constraints;
* Preconditions;
* Postconditions.
Execution feedback continuously improves future planning.
---
# Architectural Constraints
The following constraints shall always hold:
* Every Plan owns one Goal.
* Every Step belongs to one Plan.
* Steps execute according to dependency constraints.
* Planner owns planning.
* Executor owns execution.
* Knowledge Model provides planning context.
* Components communicate only through defined architectural contracts.
